What runs where — decided on the silicon you own, in under 10 ms.

HARP is an offline-capable assistant built on a calibrated routing gate. It runs each task on your device when it can, escalates to the cloud only when a task genuinely needs it, and keeps working when the network doesn't.

Under-routing is the dangerous direction — a hard task silently kept on a small model. A conformal threshold, fit on the tasks the device gets wrong, bounds it at α. Over-routing floats as the disclosed cost of today's length/keyword score; a trained encoder lowers it. Reproduce: python mac_demo/calibrate_real.py --n 300.

How a task is routed

Four steps, in order of authority. Never a raw guess.

01 · SCORE

Read the hardness

A single-pass on-device encoder rates the task's hardness u — no decode, always resident.

< 10 ms, on-device
02 · CALIBRATE

Turn score into probability

An isotonic map converts the raw score into a real edge-error probability — stable, not a wobbly margin.

isotonic (PAV)
03 · GATE

Bound the danger

A conformal threshold δ, fit on the wrong-set, keeps silent under-routing ≤ α. Escalate iff u > δ.

conformal · α-bounded
04 · GUARD

Hardware has final say

Offline, or a task the device can't run? The guard overrides everything and fails closed to local.

never overridden
→ answers LOCALorESCALATE — with a reason you can read on the X-HARP-Route header.

Why it matters

Cloud-only routers route between datacenters. Three kinds of work go unserved.

Low-connectivity

The network isn't a given

A field, a basement clinic, a moving train. A cloud-only router has nothing to fall back to when the link drops — the task just fails.

Privacy-sensitive

The prompt leaves the device

Health notes, financials, personal records. "Route to the best cloud model" still ships every token off the device.

Cost-sensitive

Every task pays rent

A greeting, an acknowledgement, a one-hop lookup — all billed at frontier prices. The cheapest task is the one that never left.

Architecture

One contract. Swap any silicon behind it.

Every backend satisfies the same interface — capabilities · infer · profile. The router negotiates capabilities; it never imports a concrete backend. A new accelerator stubs in behind the same contract.

The router never fights the hardware. Complexity decides local vs escalate; a separate capability/connectivity guard decides what's physically possible — and always wins. Offline, a queued on-device task still answers; a cloud task fails closed, no silent loss.

Run it

Routing on hardware you already own.

The canonical demo runs on your Mac's Apple Silicon plus NVIDIA Nemotron — real answers, real latencies, offline fail-closed. Mock mode needs nothing but Python.

# routing table — no models, no keys, stdlib only
$ python mac_demo/harp_demo.py --mock
# live: on-device (Ollama) + cloud (Nemotron via NIM), real answers
$ python mac_demo/harp_demo.py
$ python mac_demo/harp_demo.py --offline # every task fails closed to on-device
# reproduce the measured under-routing number
$ python mac_demo/calibrate_real.py --n 300
